# Candidate Event Review Workflow

## Purpose

Candidate event review keeps future automation separate from the approved historical analog dataset. Raw candidates can be collected or generated by a prototype workflow, but they do not become dataset rows until a human reviewer checks source quality, eligibility, coding, and documentation.

## Candidate Collection

Candidate records may come from:

- manual research notes;
- future source-monitoring scripts;
- curated source lists;
- prototype ingestion output such as `results/event_ingestion_candidates.json`.

At the candidate stage, records are not approved historical events. They should be stored separately from `data/historical_analog_events.csv`.

## Screening

Each candidate should be screened against `docs/event_collection_protocol.md`.

Screening questions:

- Does the event have a public date?
- Is the source reliable and dated?
- Is the event relevant to geopolitical competition, industrial policy, sanctions, technology restrictions, military pressure, or strategic supply chains?
- Can it be assigned a dominant event family?
- Does it add comparison value beyond existing rows?

Possible `eligibility_status` values:

- `needs_human_review`
- `eligible`
- `defer`
- `reject`

## Coding

Eligible candidates should be coded using:

- `docs/event_family_taxonomy_v2.md`
- `docs/event_coding_template.md`
- `scripts/validate_event_entry.py`

Coding must distinguish:

- state-support signals;
- restriction or pressure signals;
- qualitative market interpretation;
- observed pathway labels;
- source-based evidence notes.

## Approval or Rejection

Approval requires:

- reliable source;
- defensible event date;
- complete required fields;
- taxonomy-consistent event family;
- reviewer note explaining any uncertainty;
- no invented financial result or forecast claim.

Rejected candidates should keep a brief rejection note so the same weak candidate is not repeatedly reintroduced.

## Dataset Integration

Only approved candidates should enter `data/historical_analog_events.csv`.

Integration steps:

1. Assign a stable event ID.
2. Add a row to `data/historical_analog_events.csv`.
3. Run `python3 scripts/validate_historical_analog_dataset.py`.
4. Run `python3 scripts/run_all_checks.py`.
5. Regenerate coverage with `python3 scripts/dataset_coverage_analysis.py`.
6. Document the event batch and unresolved `TBD` fields.

## Why Human Review Is Required

Human review remains required because event coding depends on source interpretation, historical context, and judgment about dominant event family. Automated collection can help identify candidates, but it cannot determine whether a candidate is sufficiently sourced, non-duplicative, or safe to use as research evidence.

The approved dataset must remain conservative, auditable, and clearly separated from unverified candidates.
